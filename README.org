* Overview

This repository contains Logan's Emacs configuration, including:
- Emacs Lisp configuration files (~lisp/~).
- Org-mode configuration files (~org/~).
- Nix-based package management (~emacs.nix~).
- Tree-sitter grammar derivations (~nix/derivations/~).

*** Sentences in Comments

All comments should use proper punctuation and grammar.  End sentences with
punctuation, even if that sentence is in a list.  Use two spaces after a
sentence punctuation to make it easier to programmatically parse (such as vim's
sentence object).


*** Naming Functions

For functions specific to this configuration (as in, not general purpose, or
part of some other library), use the ~config--~ prefix.  You'll see a lot of
examples with ~config/~ but we want to use the community (defacto?) standard.

Any libraries that are being germinated in this repository can use a
~<lib-name>--~ form instead.



** Adding MELPA packages

If you need to add a package, you'll have to add it via [[file:./emacs.nix][emacs.nix]].  There are
sections based on what kind of package it most closely falls under.

+ completion-packages :: Anything directly related to (auto) completion.
+ editing :: Things that assist with the act of editing.  This covers vim/evil
  but also things like changing case, finding text, etc.
+ languages :: Major-modes for languages and syntaxes.
+ programs :: These are full on applications within Emacs.  ~org-agenda~,
  ~magit~ ~mu4e~, and various terminal emulators fall under this.
+ org-mode-packages :: Things specifically for ~org-mode~, which includes
  ~org-babel~ languages, ~org-mode~ exporters, and other helpers.
+ tree-sitter :: ~tree-sitter~ packages are so numerous they get their own
  section.
+ utility-packages :: Kind of a miscellaneous section.

These should be lexicographically sorted by their package name, even if it's a
Nix expression.

A brief description of what the package is, and, if relevant, why we would want
it should come before every package.

It's also okay to add commented packages and we leave a reason why we didn't
want that package.

** Adding Built-in packages

If a built-in package becomes required for something, I'd like ~emacs.nix~ to at
least document it.  To do this, take the name of the package and assume that is
the Nix package name for it, and add it just like you would for [[Adding MELPA
packages]], but comment the package out, and as part of the package's descriptive
comment, add that it is a built-in package.

This will assist in searching, and also remind us of where we should go looking
for it (not MELPA).

** Editing ~org/~ Configuration Files

Org-based configs follow a literate programming pattern.  Every ~org/~ file
uses the /noweb stitch/ pattern, which has two parts:

*** Part 1: Named blocks with ~:tangle no~

Each logical setting or feature lives in its own named source block.  All
individual blocks use ~:tangle no~ — they are /never/ tangled directly.

#+begin_src org
,#+name: config/my-feature
,#+begin_src emacs-lisp :results none :tangle no
(setq my-feature-variable t)
,#+end_src
#+end_src

*** Part 2: A ~* stitch~ block with ~:tangle yes :noweb yes~

A single ~* stitch~ (or ~* all together~) section at the bottom of the file
assembles every named block into the final output via noweb references.  This
is the /only/ block in the file with ~:tangle yes~.

#+begin_src org
,* stitch
,#+begin_src emacs-lisp :results none :tangle yes :noweb yes
<<config/my-feature>>
(use-package my-package
  :init
  <<config/my-other-feature>>
  ...)
,#+end_src
#+end_src

*** Why this pattern matters

The build system (~nix/tangle-group.el~) uses a two-pass strategy:

- Pass 1 :: Tangles only ~:tangle yes~ blocks.
- Pass 2 :: Tangles bare blocks (no ~:tangle~ keyword), but /only/ when Pass 1
  produced no output.

A file that mixes ~:tangle yes~ content blocks with bare blocks is "MIXED":
Pass 1 runs (finding the ~:tangle yes~ blocks) and Pass 2 is silently skipped,
so the bare blocks are lost.  The stitch pattern avoids this entirely — exactly
one ~:tangle yes~ block (the stitch) collects everything.

*** Exception: ~org-agenda.org~

~org-agenda.org~ is intentionally MIXED.  Some blocks use ~:tangle yes~ for
settings safe to include in the Nix build.  Bare blocks contain user-specific
paths (e.g. ~~/notes~) that must /not/ be tangled during a Nix build because
the paths don't exist at build time.  Do not convert this file to the stitch
pattern.

*** Adding a new configuration block

1. Write the block with a ~#+name:~ and ~:tangle no~:

   #+begin_src org
   ,#+name: config/my-new-thing
   ,#+begin_src emacs-lisp :results none :tangle no
   (setq my-new-thing t)
   ,#+end_src
   #+end_src

2. Reference it from the ~* stitch~ block:

   #+begin_src org
   ,#+begin_src emacs-lisp :results none :tangle yes :noweb yes
   <<config/my-new-thing>>
   ...
   ,#+end_src
   #+end_src

Within a ~use-package~ form, ~:init~ is for variable settings that must precede
the package load, and ~:config~ is for everything that runs after.  Named blocks
containing function definitions should be expanded at the top level of the
stitch block, before any ~use-package~ form, so that the functions exist by the
time ~use-package~ references them.

** Trajectory

*** ~org-mode~ vs vanilla Lisp
I initially started out with just Lisp files for configs, then I started moving
everything to ~org-mode~.  Then I found there were way too many things I wanted
bootstrapped before ~org-mode~, so I decided to just go back to Lisp.  While I
enjoy some light literate programming, ~org-mode~'s benefits do not presently
outweigh its cost (too much splicing of startup dependencies).  So just make all
new configs under ~lisp/<name>-config.el~.  Include it in the relevant section
for ~lisp/init-batteries.el~



* Usage

This flake can be imported into your NixOS, nix-darwin, or home-manager
configuration:

#+begin_src nix
{
  inputs = {
    emacs-config.url = "git+https://gitea.proton/logan/emacs-config";
    # ... other inputs
  };

  outputs = { self, emacs-config, ... }: {
    # For NixOS or nix-darwin hosts:
    darwinConfigurations.myhost = nix-darwin.lib.darwinSystem {
      modules = [
        emacs-config.darwinModules.default
        # ... other modules
      ];
    };

    # For home-manager:
    homeConfigurations.myuser = home-manager.lib.homeManagerConfiguration {
      modules = [
        emacs-config.homeModules.ssh-config-emacs
        # ... other modules
      ];
    };
  };
}
#+end_src

* Structure

- ~emacs.nix~ :: Main Emacs configuration with package definitions.
- ~lisp/~ :: Emacs Lisp configuration files.
- ~org/~ :: Org-mode configuration files for various modes and features.
- ~nix/~ :: Additional Nix modules and derivations.
  - ~derivations/~ :: Custom package derivations (e.g., ~tree-sitter-jq~,
    ~emacs-app~).
  - ~home-configs/~ :: Home Manager configuration modules.
- ~doom-emacs.org~ :: Documentation for Doom Emacs-specific configurations.

** ~emacs-app~ (macOS)

On macOS, this flake exports an ~emacs-app~ package that opens the GUI
~Emacs.app~ bundle instead of the terminal binary:

#+begin_src sh
emacs-app          # open Emacs.app
emacs-app foo.org  # open a file in Emacs.app
#+end_src

Internally it calls ~open -a Emacs "$@"~, which lets macOS route the launch
through the ~.app~ bundle so the GUI (fonts, input methods, system integration)
is used rather than the Nix store terminal binary.

To make ~emacs-app~ available on your ~PATH~, include it in your host
configuration:

#+begin_src nix
environment.systemPackages = [
  emacs-config.packages.${system}.emacs-app
];
#+end_src

This package is macOS/darwin only.
